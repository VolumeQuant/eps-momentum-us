📋 <b>EPS 시스템 진단 (쉽게 정리)</b>

━━━━━━━━━━━━━━━

<b>1. 무슨 일이 있었나</b>

MU가 4/27에 1등이었는데 4/28에 13등으로 떨어졌다.
근데 EPS 전망 자체는 거의 안 변했다 ($86.23 → $86.37).

"왜 입력은 안 변했는데 출력만 흔들리지?" — 이게 사용자 의문의 출발점.

━━━━━━━━━━━━━━━

<b>2. 시스템이 하는 일 (비유)</b>

종목을 <b>"한 달 전과 비교해서 얼마나 좋아졌는가"</b>로 매긴다.
마라톤 선수 비교하듯, 한 달 전과 지금의 페이스를 본다.

━━━━━━━━━━━━━━━

<b>3. 진짜 원인</b>

3월 18일에 MU 어닝 발표했고, 그날 EPS 전망이 폭등했다 ($46 → $80, +73%).

이게 한 달 후인 4월 말에 시스템 비교에 영향을 준다:

🟦 <b>4/27이 보는 "한 달 전"</b> = 3월 16일경
   = 어닝 발표 <b>직전</b>
   = 그때 EPS 전망 = $46 (낮음)
   = "지금 $86" vs "한 달 전 $46" = <b>엄청 좋아짐 → 1등</b>

🟥 <b>4/28이 보는 "한 달 전"</b> = 3월 17일경
   = 어닝 발표 <b>직후</b>
   = 그때 EPS 전망 = $80 (이미 높음)
   = "지금 $86" vs "한 달 전 $80" = <b>별 차이 없음 → 13등</b>

<b>단 하루 차이로 비교 기준점이 어닝을 넘어가면서 순위가 폭락.</b>
이게 핵심.

━━━━━━━━━━━━━━━

<b>4. 두 가지 문제로 나눠야 함</b>

🟢 <b>의도된 동작 (못 고침)</b>
어닝 같은 큰 이벤트를 시스템이 점수에 반영하는 것 자체는 원래 디자인 목적. "한 달 전과 비교"라는 측정 방식의 자연스러운 결과. 어닝 효과를 무시하면 시스템 의미가 사라짐.

🔴 <b>고칠 수 있는 부작용</b>
이 와중에 부수적으로 점수가 -43% 폭락 (adj_score 136 → 78).
이건 의도 아니라 "EPS 변화 4구간 중 한 구간이 +100%로 cap에 걸리면서 가속도 계산이 부호 반전"하는 부작용. 노이즈.

━━━━━━━━━━━━━━━

<b>5. 내가 헛돌았던 이유</b>

처음엔 두 문제를 한 덩어리로 봐서 자꾸 잘못된 fix 시도했음.
사용자가 구분 짚어줘서 깨달음:
"어닝 효과 반영"은 의도, "점수 폭락"은 부작용.

또 DB에 모든 데이터 매일 저장돼있는데 yfinance 다시 부르려다 사용자 시간 낭비시킨 점 사과.

━━━━━━━━━━━━━━━

<b>6. 어떤 fix가 가능한가</b>

🔧 <b>가능한 fix (γ 또는 γ'')</b>:
부작용(점수 -43% 폭락) 차단.
adj_score: 78 → <b>112</b>로 회복.
사용자 본질 우려 ("입력 안정 → 출력 흔들림") 일부 해결.

⚠️ <b>한계</b>:
순위(cr) 자체는 여전히 1→13 정도로 변함. 이건 어닝 효과(의도) 부분이라 못 고침.
순위 폭락의 95%는 의도된 동작이고, 5%만 부작용.

━━━━━━━━━━━━━━━

<b>7. "왜 점수 회복(78→112)했는데 순위는 안 회복?"</b>

사용자가 가장 궁금해한 부분.

답: <b>순위는 점수로 매기는 게 아님</b>. 가격/EPS 가성비(=adj_gap)로 매김.

MU 4/28: 가격이 EPS보다 빠르게 오르는 중 (가격 7일간 +12%, EPS 1%). 가성비 양수 = 비싸짐 = 순위 후순위. 점수와는 다른 차원.

이건 시스템 설계 자체가 "가성비"를 순위 기준으로 삼아서 그런 것. 점수가 좋아도 가성비 안 좋으면 순위 떨어짐.

━━━━━━━━━━━━━━━

<b>8. 결정 옵션</b>

<b>A) 부작용만 차단 (γ'') → 메인 워크플로우 재가동</b>
• 점수 폭락(-43%)은 막아짐
• 순위는 거의 그대로
• 안전, 보수적

<b>B) 시스템 디자인 자체 바꾸기</b>
• 점수가 좋으면 순위도 좋아지게 식 자체 변경
• 큰 작업, 백테스트 다시 필요
• 위험 있음

<b>C) 그대로 두고 재가동</b>
• 시스템 의도 받아들임
• fix 안 함

━━━━━━━━━━━━━━━

<b>9. 추천</b>

A부터 가는 게 맞다고 본다.
이유:
• 부작용은 명확하니 잡고
• 순위 결정 식 변경(B)은 백테스트 다시 필요해서 시간 많이 듦
• 메인 워크플로우 너무 오래 disabled 둘 수 없음

A 적용해서 재가동 → 그 후 B 검토는 별도 작업으로.
