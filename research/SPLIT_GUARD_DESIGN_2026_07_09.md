- **권장안 = (a) 감지-보정형**(하루 가격비 시그니처 감지 → `yfinance Ticker.splits`로 확인 → 조정계수를 리플레이 read-time에 적용). 원천-보정형(DB 과거행 일괄 rewrite)은 blast radius가 크고(ma60/ma120/high30도 같이 재계산해야 정합) 대부분의 유니버스(보유 안 하는 1,400여 종목)엔 불필요한 비용이라 기각.
- **오염 표면 8곳** 식별(활성 5 + 죽은코드 3). 그 중 **`high30`(dd_30_25 진입필터)은 지금 이 순간(2026-07-09) MLI에서 실제로 오작동 중** — 07-01 2:1 분할 후 high30이 분할 전 최고가($139.5)에 멈춰 dd30=-60%로 계산되어 최소 29거래일(~8/12까지) 진입 차단.
- **rev90(ntm_90d 등 EPS 추정 시계열)은 오염되지 않음** — MLI 분할일 실측으로 확인: `ntm_current/7d/30d/60d/90d`가 같은 날 한 번에 전부 동일 비율로 리스케일됨(야후 eps_trend가 price처럼 매 호출마다 전체 창을 최신 주식수 기준으로 재조정). 문제는 **오직 "저장된 과거 raw price를 오늘 raw price와 직접 비교"하는 지점**뿐.
- 판정일 구현 규모: **daily_runner.py диff 약 250~350줄**(신규 헬퍼 모듈 + 5개 호출부 통합), 위험도 **중간**(핵심 NAV/리플레이 함수를 건드리므로 "분할 없는 케이스=완전 동일 숫자"가 1차 합격 기준).

### 1.1 실측: MLI 라이브 버그 (지금 진행 중)

```
DB ntm_screening (ticker='MLI')
2026-06-30  price=122.93  high30=139.51
2026-07-01  price=57.42   high30=139.51   ← 2:1 분할(야후 Ticker.splits 확인: 2026-07-01, ratio=2.0)
2026-07-07  price=55.67   high30=139.51   ← high30이 분할 전 최고가에 그대로 고정
```
`dd30 = (55.67-139.51)/139.51 = -60.1%` → `daily_runner.py` v84 dd_30_25 필터(`< -25%`)에 걸려 **MLI는 실제로는 정상 주가 흐름인데도 앞으로 최대 29거래일(그 최고가가 DB `LIMIT 29` 윈도우 밖으로 밀려날 때까지, 대략 8/12경)간 신규 진입이 원천 차단된다.** 같은 코드 경로가 6/25 dd_30_25 도입 당시엔 상승장 전용이라 "실보험, 비용 0"으로 검증됐지만, 그 검증엔 분할 케이스가 없었다 — 이번이 첫 실전 사례.

대조군(자기치유 확인): CAR/KD/ASGN/POWL은 분할이 29거래일 이상 지나 `high30` 윈도우에서 이미 빠져나가 지금은 정상.

### 1.2 실측: rev90(EPS 추정 시계열)은 오염되지 않음

```
MLI ntm_current/7d/30d/60d/90d
2026-06-30:  8.574 / 8.567 / 8.545 / 7.714 / 7.636
2026-07-01:  4.287 / 4.284 / 4.273 / 3.858 / 3.819   ← 전부 정확히 ~0.5배(2:1 분할과 일치)
```
**결론**: 야후 `eps_trend`는 price의 `auto_adjust`와 동일하게, **호출 시점 기준으로 과거 창(90일 전 추정치 포함) 전체를 그 시점의 최신 주식 수 기준으로 재조정해서 돌려준다.** 즉 `min_seg`/`eps_chg`/`adj_gap`/게이트(`gap = ntm_current/TTM`)는 **같은 날 저장된 `ntm_current`와 `ntm_90d`를 비교하므로 항상 내부적으로 정합**하다 — DB에 저장된 서로 다른 날짜의 ntm_* 값을 직접 비교하는 코드가 없는 한(실제로 없음, 전부 같은 행에서 계산) 스플릿 문제와 무관하다. 사용자가 우려한 "n90 오염 가능성"은 **기각** — 단, 이는 "yfinance가 계속 이렇게 동작한다"는 벤더 의존 가정이며 향후 회귀 시 재확인 필요.

## 2. 가드 설계 2안 비교

### (a) 감지-보정형 — **권장**

read-time(리플레이/NAV 계산 시점)에 다음을 수행:
1. 연속 저장일 가격비가 시그니처 구간(기본 `ratio ≤ 0.75 또는 ≥ 1.34`)이면 후보로 표시.
2. 후보 날짜 ± 며칠(기본 7일, 아래 실패모드 참고) 안에 `yfinance Ticker(ticker).splits`에 이벤트가 있으면 그 비율로 조정계수 확정.
3. 조정계수 = `1/누적비율`을 그 날짜 **이전**의 모든 저장가에 곱해 "현재 주식수 기준"으로 정규화(야후 auto_adjust와 동일한 방향의 계산).
4. 확인 안 되면(§2.3 실패모드) **factor=1.0 유지 + "미확인" 목록에 기록**(무조건 보정하지 않음 — 안전 우선, 알림용).

**장점**: DB 원본 불변(감사 추적 보존), 필요한 종목·필요한 순간에만 API 호출(현재 보유 2슬롯 + 매수후보 상위 몇 종목뿐이라 비용 작음), 롤백이 코드 revert 하나로 끝남(DB migration 불필요), 기존 "forward-only"/"킬스위치" 배포 철학과 정합.

**단점/실패모드**:
- **yf splits 데이터 누락**: 실측 확인 — ASGN(2026-04-22→23, ratio 0.483)과 CAR(같은 날, ratio 0.516) 둘 다 DB엔 분할급 절벽이 있는데 `Ticker.splits`엔 해당 이벤트가 없음(ASGN은 아예 "possibly delisted" 에러). 즉 **"진짜 분할처럼 보이는데 야후가 확인 안 해주는" 케이스가 실제로 존재한다.** 이런 경우 안전한 기본값은 "보정 안 함 + 사람에게 경고"이지 "비율로 추정해서 강제 보정"이 아니다(진짜 –48% 폭락일 수도 있으므로 오히려 보정하면 진짜 손실을 지워버리는 재앙이 됨).
- **날짜 어긋남**: POWL 실측 — DB 절벽은 2026-04-01→04-02인데 `Ticker.splits`가 기록한 ex-date는 04-06(4일 차이). 정확한 날짜 매칭(`==`) 대신 **윈도우 매칭(±7일)** 이 필수.
- **배당락 오인**: 이 유니버스(시총 대형/거래대금 $1B+ 주도주)의 분기 배당은 보통 주가의 1~2% 이내 → 시그니처 임계값(±25~34%)에 전혀 안 걸림. 사실상 비이슈로 확인(설계상 임계값을 완만한 배당보다 훨씬 넓게 잡아 자동으로 배제됨).
- **역분할 오인 없음**: 비율 상/하한 양쪽을 다 스캔하므로 역분할(예: DD 2026-06-24, ratio 0.333≈1-for-3, 주가 3배 급등)도 같은 로직으로 처리됨(실측 확인).
- **BT 정합 영향**: 보정을 켜면 과거 BT 숫자가 살짝 바뀔 수 있다(스플릿을 겪은 종목이 과거에 보유됐었다면). 현재 19종목 보유이력엔 스플릿이 전혀 없음(재확인 완료) → **오늘 켜도 과거 트랙레코드(+XXX%) 숫자는 불변**이 1차 합격 기준.

### (b) 원천-보정형 (DB 과거행 일괄 조정 + 백업)

수집 시 분할 감지 → 해당 티커의 DB `price`(및 파생: `ma60`/`ma120`/`high30`) 과거 행을 전부 소급 재계산 → 백업 테이블 보관.

**장점**: 한번 고치면 모든 다운스트림 코드(현재 8곳 + 향후 신규 코드)가 자동으로 안전해짐. "raw price" 개념이 사라지고 DB 자체가 조정가 기준이 됨.

**단점**: 
- 다시 계산할 컬럼이 `price` 하나가 아니라 `ma60/ma120/high30` 전부 — 안 하면 price만 조정되고 나머지는 옛 스케일로 남아 새로운 불일치가 생김(오히려 악화). 이 3개는 각자 다른 날의 롤링 윈도우라 소급 재계산이 간단하지 않음(120일 윈도우가 분할 전후를 걸치는 경우 등).
- 유니버스 전체(1,400+ 종목 × 최대 103일)에서 분할은 극소수(스캔 결과 6건)인데, "감지→확정→쓰기"를 매 수집 사이클 전종목에 걸쳐 돌려야 함 — 비용 대비 편익 낮음(보정이 실제로 의미 있는 건 "보유했거나 보유할 가능성 있는 종목"뿐).
- 되돌리기 어려움(백업에서 복원해야 함), 감사 추적(원본 raw price가 무엇이었는지)이 깨짐, 향후 "진짜 분할이었나 재검증" 자체가 불가능해짐(원본을 지웠으므로).
- DB 변경은 이번 작업 범위(★프로덕션·DB 변경 금지) 밖이라 이 안은 애초에 이번 판정일 후보에서 실행 불가 — 비교를 위해서만 기술.

**판정**: (a) 채택. (b)는 "미래에 분할이 훨씬 잦아지고 코드 지점이 8곳보다 훨씬 늘어난다면" 재고할 옵션으로 메모만.

### 2.3 임계값 근거 (실측 기반)

- `ratio ≤ 0.75 또는 ≥ 1.34`: 3-for-2 분할(0.667)까지 잡히면서, 이 유니버스의 정상적인 어닝 서프라이즈발 하루 변동(대형주 기준 보통 ±20% 이내, 극단치 –20%대)과는 여유를 두고 구분. 임계값은 "API 호출 여부를 결정하는 값싼 스크린"일 뿐 최종 판정이 아니므로(진짜 확정은 `Ticker.splits`가 함) 다소 넓게 잡아도 비용은 API 호출 1회뿐 — false positive 걱정보다 false negative(진짜 분할을 놓치는 것) 방지에 우선순위.
- 확인 윈도우 ±7일: POWL 사례(4일 차이)에 여유를 더한 값.

## 3. 코드 초안

`research/split_guard_draft_2026_07_09.py` — 독립 실행 가능(단위테스트 3+2종 포함: 2:1 분할·10:1 분할·2% 배당락 무시·미확인 절벽(ASGN/CAR형)·날짜 어긋남(POWL형)).

핵심 함수:
- `detect_split_candidates(price_series) -> list[(prev_date, date, ratio)]` — 값싼 1차 스크린.
- `confirm_split_ratio(ticker, candidate_date, splits_fetcher, window_days=7) -> float|None` — 야후 확인(주입 가능한 fetcher로 테스트 시 합성 데이터 사용).
- `compute_split_adjustments(ticker, price_series, splits_fetcher) -> (adjustments: dict[date->factor], unconfirmed: list[date])` — 메인 진입점. **DB에 아무것도 쓰지 않고**, 호출자가 가진 `{date: price}` dict에 대해 순수 계산만 수행.
- `build_adjusted_lookup(pxh: dict[ticker][date] -> price, splits_fetcher) -> 동일 구조` — `_vm_paper_state`의 `px_map`, `_replay_holdings`의 `pxh`와 **동일한 딕셔너리 구조**로 반환 — 실제 통합 시 해당 변수를 이 함수 리턴값으로 교체하기만 하면 되도록 설계(교체 지점: daily_runner.py L3964-3968, L4232-4234, L6039-6047 부근의 `all_prices`/`daily_data` 가격 로드부).
